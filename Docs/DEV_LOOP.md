# DragonChessLegends — Continuous Dev Loop (Steam Target)

## North Star
做出“金铲铲（构筑+运营） × 杀戮尖塔/小丑牌（Roguelike决策密度）”的单人可重复游玩产品，达到可上架 Steam 的商业质量。

---

## Hard Rules
1. **只有在非常确定时才 commit**（通过自测 + 关键流程回归 + 体验检查）。
2. 每个阶段必须有：
   - 可运行版本
   - 回归清单
   - 已知问题清单
   - 下一阶段明确目标
3. 先稳定架构，再扩内容，再打磨手感与商业化包装。

---

## Iteration Pipeline (always-on)

### Stage A — 架构稳定化（当前）
目标：把 `RoguelikeFramework.cs` 的“单文件全能逻辑”拆成可维护结构。

- A1: 拆分职责（Flow/Economy/Battle/Synergy/UI）
- A2: 引入轻量配置数据层（数值和掉落权重可调）
- A3: 增加开发期开关（快速开局、跳关、战斗加速）

验收：
- 玩法表现不倒退
- 关键回归全绿（开局、商店、布阵、战斗、海克斯、结算）

### Stage B — 核心玩法闭环增强
- B1: 完整升星链（1→2→3）
- B2: 玩家生命值与失败惩罚
- B3: 战后奖励 3 选 1（金币/单位/遗物或强化）
- B4: 关卡节奏曲线重做（普通/精英/Boss节奏）

### Stage C — 内容深度（可卖点）
- C1: 单位扩充到可构筑规模（先达成 21 单位框架）
- C2: 羁绊与海克斯做可组合爆发点
- C3: 事件房/商店房/Boss机制差异化

### Stage D — 体验与商业化质量
- D1: UI/反馈/信息层级优化
- D2: 新手引导和可读性
- D3: 性能优化、存档、设置项
- D4: Steam 上架素材准备（胶囊图、描述、截图、预告片脚本）

---

## Mandatory Regression Checklist
每轮改动后至少验证：
- 能从开局走到至少第 3 关
- 商店购买/刷新/经验消耗正确
- 拖拽上阵/下场/出售不出错
- 战斗能稳定结束，不出现卡死
- 海克斯选择能正确生效
- 结算与下一关流转正确
